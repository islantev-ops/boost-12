# Проверка №1: оживить географию сервера

**Дата:** 2026-07-16
**Статус:** дизайн согласован, готов к плану реализации
**Скоуп:** пункт 1 чек-листа «Данные не уходят за границу» + починка `npm run check-site`

---

## 1. Проблема

Проверка №1 никогда не выдаёт `ok`. На контрольной выборке из 5 реальных
Битрикс-сайтов она дала «требует ручной проверки» **5 раз из 5**.

Причина не в нехватке данных, а в самой конструкции. В [`checks.ts:271-276`](../../../1-2-MVP/product-mvp/src/lib/checks.ts)
есть фактор «Физическое размещение сервера в РФ» с захардкоженным `vote: 'unknown'`:
геоисточники в MVP не подключили, и вместо реализации поставили заглушку.
А `byPresence` ([`checks.ts:59-63`](../../../1-2-MVP/product-mvp/src/lib/checks.ts))
роняет проверку в `manual` при любом `unknown`, если нет прямого нарушения.

Итог: сайт с чистым кодом, без Google Analytics и без GTM, получает «вручную»
вместо честного «счётчиков не найдено». Вечно-неизвестный фактор отравляет
рабочую проверку.

Лендинг при этом помечает пункт как **[Авто]** и обещает «Сервер в РФ». То есть
мы реализуем обещание, данное с самого начала, а не меняем состав проверок.
**Правка лендинга не требуется.**

## 2. Правовая основа

Дословно, [152-ФЗ ст. 18 ч. 5, КонсультантПлюс, ред. от 24.06.2025](https://www.consultant.ru/document/cons_doc_LAW_61801/cbf4e15b7c330f9372e876cdf2bc928bad7950ef/#dst14):

> «При сборе персональных данных, в том числе посредством
> информационно-телекоммуникационной сети "Интернет", запись, систематизация,
> накопление, хранение, уточнение (обновление, изменение), извлечение
> персональных данных граждан Российской Федерации **с использованием баз
> данных, находящихся за пределами территории Российской Федерации**, не
> допускаются, за исключением случаев, указанных в пунктах 2, 3, 4, 8 части 1
> статьи 6 настоящего Федерального закона».

Проверено по процедуре скилла `ispolzovat-konsultant`: страница статьи открыта
дважды, редакция подтверждена по оглавлению закона.

**Объект нормы — база данных, а не веб-сервер.** Это важно для формулировок
(§5): факт, который мы измеряем, и норма, на которую ссылаемся, — разные
уровни, и в отчёте они не должны сливаться.

## 3. Допущение о со-расположении базы данных

**Допущение:** у сайта целевого сегмента база данных находится там же, где
веб-сервер. Обе либо в РФ, либо за её пределами.

**Основание:** типовой Битрикс-сайт малого и среднего бизнеса — это один
хостинг, где PHP и MySQL стоят на одной машине. Решение владельца продукта,
согласовано 2026-07-16. Оно согласуется с позицией лендинга, который прямо
называет заграничный хостинг одним из трёх способов нарушить требование
локализации.

**Чем платим:** сайт с фронтендом за границей и базой в РФ получит ложное
обвинение. Допущение принято сознательно, зафиксировано здесь, чтобы через
полгода происхождение вердикта было видно без археологии.

## 4. Дизайн

### 4.1. Границы проверки

Проверка №1 остаётся одним пунктом чек-листа и получает два фактора.
Третий — «локализация БД» — **удаляется**: он и был той заглушкой, что убивала
проверку. Отдельной строкой в «Требует ручной проверки» он не появляется:
по допущению §3 география сайта отвечает и за базу.

| Фактор | Голоса | Источник |
|---|---|---|
| А. Иностранные счётчики (GA/GTM) | `violation` / `ok` | код страниц; **работает, не трогаем** |
| Б. Размещение сайта | `violation` / `ok` / `unknown` | DNS → IP → RDAP (+ второй источник) |

Вердикт пункта собирается прежним `byPresence`: любое `violation` → нарушение;
иначе любое `unknown` → вручную; иначе `ok`.

### 4.2. Источники и асимметрия

**Источник А — RDAP RIPE**, `https://rdap.db.ripe.net/ip/<IP>`. Бесплатно, без
ключа, без регистрации, без лимитов, без ограничений на коммерческое
использование. Отдаёт `country` и `name` (netname) — то есть страну и хостера.
Проверен на всех пяти сайтах выборки: 5/5 определились однозначно.

**Источник Б — второй геоисточник.** Дёргается **только когда источник А сказал
«не РФ»**.

Асимметрия — прямое следствие требования PRD «ноль ложных срабатываний»:

- **RDAP говорит RU** → фактор `ok`. Ошибка здесь дала бы пропуск нарушения, а
  не ложное обвинение. Второй источник не нужен.
- **RDAP говорит не RU** → это заявка на обвинение, её обязан подтвердить
  независимый источник. Сошлись → `violation`. Разошлись → `unknown`.
- **netname опознан как CDN** → `unknown`. За CDN происхождение не видно.

Практическое следствие: заграничных сайтов в сегменте единицы, поэтому
источник Б вызывается несколько раз в неделю, а не на каждый аудит. Любой
бесплатный тариф это выдерживает.

**Источник Б — ipwho.is**, `https://ipwho.is/<IP>`. Выбран по итогам проверки
всех кандидатов вживую:

- ключ и регистрация не нужны — *«No signup, no API key required»*;
- 1 000 запросов в сутки бесплатно;
- **коммерческое использование разрешено прямо** — *«Commercial use allowed»*
  ([тарифы](https://ipwhois.io/pricing), [документация](https://ipwhois.io/documentation));
- проверен на живых адресах: `31.31.198.246` → `country_code: RU, city: Moscow`;
  `8.8.8.8` → `country_code: US, city: Mountain View`. Чистый JSON с полем
  `success`.

Оговорка из их же условий: *«No SLA. Uptime is not guaranteed»*. Для нас это не
проблема — недоступность даёт `unknown` → «вручную» (§6).

Отвергнутые кандидаты (все проверены, а не отброшены по памяти):

- **2ip (`api.2ip.me`)** — ключ не нужен, но бесплатный лимит исчерпывается
  мгновенно: первый запрос `geo.json` вернул 200, **второй подряд — HTTP 429**
  «The limit of returned objects has been reached», хотя документация обещает 10
  запросов в сутки. Вдобавок тело ошибки 429 приходит **HTML-страницей, а не
  JSON**. Как источник подтверждения обвинения непригоден.
- **ip-api.com** — бесплатный тариф прямо запрещает коммерческое использование:
  *«The use of the API is strictly limited for a non-commercial purpose and in a
  non-commercial environment»* ([условия](https://ip-api.com/docs/legal)).
  Лидогенерация для агентства — коммерция. Не подходит.
- **MaxMind GeoLite2** — требует регистрации, лицензионного ключа, атрибуции и
  удаления старой базы в течение 30 дней после выхода новой; сам MaxMind
  указывает на пониженную точность бесплатной City-базы
  ([условия](https://dev.maxmind.com/geoip/geolite2-free-geolocation-data)).
  Возни больше, чем пользы, при наших объёмах.
- **Парсинг страницы 2ip.ru** — отдаёт `HTTP 503` на программный запрос
  (проверено дважды). В браузере открывается, боту — нет.

**Запасной вариант**, если ipwho.is начнёт подводить: RIPEstat
(`https://stat.ripe.net/data/maxmind-geo-lite/data.json?resource=<IP>`) —
бесплатно, без ключа, проверен, отвечает `status: ok`. Держим в уме, в код
сейчас не тащим.

Недоступность источника Б **не блокирует аудит**: она даёт `unknown` → пункт
уходит в «вручную». Молчаливое падение в `ok` или `violation` недопустимо.
Ответ с `HTTP != 200` или без ожидаемого поля — это `unknown`, а не повод
парсить тело как JSON.

### 4.3. Устройство кода

Новый модуль `src/lib/geo.ts` — единственное место, знающее про сеть и внешние
источники:

```
resolveHosting(url) → {
  ip: string,
  country: string | null,     // ISO-код из RDAP
  netname: string | null,     // имя сети/хостера
  isCdn: boolean,
  confirmedBy: string[],      // какие источники сошлись
  error?: string,
}
```

`checks.ts` про сеть и про RDAP не знает: получает готовый факт и раздаёт
голоса. Границу держим ради тестируемости — фактор Б проверяется на
фикстурах, без обращения к сети.

Список CDN-netname выносится в константу модуля и комментируется: почему
каждая запись там, и что она означает.

## 5. Формулировки в отчёте

Правило: печатаем **только измеренный факт** и норму близко к тексту. Про базу
данных клиента не пишем ничего — ни утверждения, ни вопроса. Мы её не видели;
всё, что мы знаем, — где стоит сайт. Заграница сама по себе красный флаг, и
этого для письма достаточно: связку «сайт за границей → а где база?» владелец
достроит сам, а мы не подставимся под возражение «база у нас в Москве, вы
соврали».

**Сайт в РФ, счётчиков нет** → `ок`:
> «Google Analytics и Google Tag Manager не обнаружены. Сайт размещён в РФ:
> IP 31.31.198.246, хостер REG.RU, страна RU по данным RIPE».

**Сайт за границей** → `нарушение`:
> «Сайт размещён за пределами РФ: IP <адрес>, страна <код>, хостер <netname> —
> подтверждено двумя независимыми источниками. Закон запрещает запись,
> накопление и хранение персональных данных граждан РФ с использованием баз
> данных, находящихся за пределами территории РФ (152-ФЗ, ст. 18 ч. 5).»

И на этом всё. Первое предложение — то, что мы измерили и что владелец
перепроверит за минуту. Второе — норма близко к тексту, со ссылкой с якорем.
Ни одного утверждения о его базе данных мы не делаем.

Со-расположение базы (§3) остаётся основанием **вердикта**, но в текст отчёта
не попадает: мы его не измеряли. Вердикт на допущении строится, отчёт на нём —
нет.

**За CDN** → `вручную`:
> «Сайт отдаётся через CDN (<netname>), физическое размещение снаружи не
> определяется. Требуется ручная проверка».

В каждом случае в отчёт идут IP, netname и страна: владелец перепроверяет за
минуту.

## 6. Граничные случаи

| Ситуация | Поведение |
|---|---|
| DNS не резолвится | `unknown` → вручную. Сайт при этом мог открыться через прокси — молча не роняем. |
| Несколько A-записей | Проверяем все. Все в РФ → `ok`. Хоть одна за границей → как «не РФ» (нужен источник Б). |
| RDAP недоступен / таймаут | `unknown` → вручную. Не `ok`. |
| RDAP не вернул country | `unknown` → вручную. |
| Источник Б недоступен, когда нужен | `unknown` → вручную. Обвинение без подтверждения не выпускаем. |
| Источники разошлись | `unknown` → вручную. |
| IP приватный/зарезервированный | `unknown` → вручную. |
| Сайт за CDN | `unknown` → вручную, независимо от страны CDN-узла. |

Общее правило: **любая неопределённость → `manual`**, никогда не `ok` и никогда
не `violation`. PRD §8: тихого четвёртого исхода быть не должно.

## 7. Починка `npm run check-site`

Команда из [README.md](../../../README.md) не запускается: скрипт вызывает
`tsx`, а `tsx` не значится в зависимостях. Падает с «tsx не является внутренней
или внешней командой». Обход — `npx tsx`.

Правка: добавить `tsx` в `devDependencies` в
[`package.json`](../../../1-2-MVP/product-mvp/package.json).

## 8. Вне скоупа

- **Англицизмы (блок Б).** Вторая мёртвая проверка, отложена сознательно.
- **Ложное срабатывание в проверке №4.** Подтверждено на yarkremlin.ru:
  инструмент заявил «состав согласия неполон», хотя в документе есть и цель, и
  перечень действий, и способ отзыва. Три бага в регексах
  ([`checks.ts:404-409`](../../../1-2-MVP/product-mvp/src/lib/checks.ts)) плюс
  корневая причина — `$('body').text()` в
  [`crawl.ts:109`](../../../1-2-MVP/product-mvp/src/lib/crawl.ts) склеивает
  слова на границах тегов («исключительно с целью» → «исключительнос целью»).
  Отдельная задача, живой баг.
- **Глубина обхода.** 3–4 страницы при `MAX_PAGES = 18`.
- **Скриншоты-пруфы** из PRD §5.4.

## 9. Критерии готовности

1. На выборке из 5 сайтов (`yarcmyk.ru`, `magic-yarn.ru`,
   `zapovednaya-polyana.ru`, `yarkremlin.ru`, `we-energy.ru`) проверка №1 даёт
   `ок` — вместо нынешних 5/5 «вручную». Это и есть «оживить».
2. Есть тест на фактор Б на фикстурах, без сети: РФ → `ok`; заграница при
   согласии источников → `violation`; расхождение → `unknown`; CDN → `unknown`;
   таймаут → `unknown`.
3. Ни один путь не приводит к `ok` или `violation` при недоступном источнике.
4. В отчёте по каждому вердикту есть IP, netname и страна.
5. `npm run check-site https://example.ru` работает после `npm install`.
6. Формулировка нарушения содержит норму близко к тексту и ссылку с якорем на
   КонсультантПлюс.

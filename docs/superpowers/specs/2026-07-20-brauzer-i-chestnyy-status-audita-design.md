# Аудит через реальный браузер + честный статус защиты

Дата: 2026-07-20
Статус: черновик на ревью

## Задача

Аудит сайта на комплаенс сейчас скачивает страницы простым `fetch` из
`src/lib/crawl.ts`. На сайтах под антибот-защитой (KillBot, Cloudflare,
DDoS-Guard) это возвращает не сайт, а страницу-заглушку защиты. Аудит
принимает заглушку за сайт, из-за чего:

1. **Выдаёт ложные нарушения.** Проверка находит счётчики/трекеры в коде
   самой заглушки и приписывает их клиенту.
2. **Прячет остальное в «вручную».** Проверки видят пустой каркас и уходят
   в ручной режим с формулировкой про «невидимый подвал», хотя настоящая
   причина — защита.

Пользователь ("часто — это типичный клиент") подтверждает, что защищённые
сайты — не исключение, а норма.

### Проверено фактом (2026-07-20, https://www.rustehnika.ru/)

| Способ получения страницы | Результат |
|---|---|
| Текущий `fetch` (Chrome-UA) | заглушка KillBot, HTTP 200, `text/html` |
| Playwright **headless** | не прошёл: 26 с «Проверка пользователя…» |
| Playwright **headed** (видимый браузер) | **прошёл за ~19 с — реальный сайт** |

Настоящий сайт rustehnika.ru **не содержит** Google Analytics
(`googletagmanager` = 0 на реальной странице). Текущий аудит выдал по нему
«НАРУШЕНИЕ: установлен Google Analytics» — пруф указывал на код KillBot
(`script.src = googletagmanager… ; id = kbsmKi`, где `kb` = KillBot).
Это ложное обвинение, готовое уйти клиенту в письме.

## Границы этой работы

Реализуем **шаг 1 из двух**, выбор пользователя ("сначала самое больное:
браузер + не врать"):

**В этой спеке:**
- Получение страниц через реальный (видимый) браузер вместо `fetch`.
- Честный статус «сайт закрыт защитой, автопроверка невозможна», когда
  даже браузер не прошёл.
- Фильтр от чужих скриптов: не засчитывать трекеры/маркеры из кода
  антибота как нарушения клиента.

**НЕ в этой спеке (отдельный шаг 2, позже):**
- Обход всего сайта (глубина > 1, шаблонная выборка страниц, потолок
  ~500 загрузок). Пока сохраняется текущая логика отбора страниц
  (главная + ссылки по хинтам + перебор известных путей, `MAX_PAGES`).
- Фоновая задача со статусом `crawling` и опросом прогресса из UI.
- Таблица `pages` для воспроизводимости аудита.

Обоснование деления: браузер и «не врать» закрывают активный риск ложных
обвинений и работают в рамках текущей синхронной архитектуры. Полный
обход требует перевода аудита в фоновый режим — большая отдельная переделка.

## Как это будет работать

### Компонент: браузерный загрузчик страниц

Новый модуль (рабочее имя `src/lib/browser.ts`) на Playwright/Chromium.
Единственная ответственность: по URL вернуть готовый (отрисованный) HTML
и финальный URL, пройдя антибот-челлендж, либо честно сообщить, что не
прошёл. Интерфейс наружу — та же форма, что сейчас возвращает `fetchPage`
в `crawl.ts` (`{ url, status, html, text }`), плюс поле результата
челленджа. За счёт совпадения формы `crawl.ts` меняется точечно: заменяем
источник HTML, а логику отбора страниц и сборки снапшота не трогаем.

Поведение загрузчика на одну страницу:
1. Открыть URL в видимом браузере (headed под xvfb на сервере).
2. Дождаться прохождения челленджа: опрос до ~30 с, признак «прошли» —
   исчезновение маркеров антибота (заголовок вида «Проверка
   пользователя…» / «user verification», скрипты `window.kb*` и т.п.
   сменились реальным контентом). Точный детектор — по наблюдаемым
   маркерам, вынести в отдельную функцию, покрыть тестом на сохранённых
   HTML заглушки и реального сайта.
3. Если за таймаут маркеры не исчезли → вернуть результат «заглушка,
   не прошли».
4. Вернуть отрисованный `document` (`page.content()`), финальный URL.

**Позиция по обходу защиты (важно, зафиксировать).** Мы используем
_настоящий_ браузер без маскировки — то же, что делает человек, открывая
сайт мышкой: выполняем JS, принимаем куки, ждём. Мы **не** внедряем
ротацию прокси, подмену отпечатка (stealth), решалки капчи, эмуляцию
живого пользователя. Это осознанная граница: (а) продукт продаёт
юридический комплаенс — собирать доказательства, обходя техзащиту чужого
сайта, подрывает саму позицию; (б) гонка с обновлениями антибота
бесконечна; (в) риск бана серверного IP шире одного сайта. Если honest
headed-браузер сайт не прошёл — это валидный исход «проверка невозможна»,
а не повод усиливать обход.

### Изменение: честный статус в снапшоте

В `SiteSnapshot` (`src/lib/types.ts`) добавить признак, что сайт закрыт
защитой и содержимое недостоверно (рабочее имя `blockedByAntibot`).
Заполняется в `crawlSite`, если браузерный загрузчик не прошёл челлендж на
главной.

В `auditSite` (`src/lib/audit.ts`) добавить ветку по аналогии с
существующей `!reachable`: если `blockedByAntibot`, вернуть снапшот без
`findings` (проверки не запускаем — им нечего проверять) и без письма.
Сейчас `route.ts` уже не генерирует письмо для недоступного сайта — тот же
принцип.

В отчёте/карточке аудита этот статус показывается явным текстом («Сайт
закрыт защитой от автоматического доступа. Автоматическая проверка
невозможна, требуется ручная проверка в браузере»), а не как список
нарушений и не как 9 пунктов «вручную».

### Изменение: фильтр от чужих скриптов

Даже когда браузер прошёл защиту, в истории навигации мог остаться код
заглушки. Проверки трекеров/аналитики в `src/lib/checks.ts` (проверка 1,
строки ~370–395: `google-analytics.com`, `googletagmanager.com`, GTM,
Метрика) должны засчитывать маркер, только если он присутствует в
**финальном отрисованном DOM реального сайта**, а не в промежуточной
заглушке. Практически: детектор антибота помечает, какие скрипты/маркеры
принадлежат прослойке защиты (домен-источник, префиксы вроде `kb*`), и
проверка их игнорирует. Покрыть тестом на сохранённом HTML заглушки
rustehnika (там `googletagmanager` с `id=kbsmKi` не должен давать
нарушение) и на реальном DOM (там, где GA действительно есть, — должен).

## Инфраструктура и риски

- **Ресурсы VPS (блокирующая проверка ДО реализации).** Chromium ест
  200–500 МБ на вкладку плюс системные библиотеки и xvfb. Пользователь
  выбрал «тот же VPS + Chromium + xvfb». Перед реализацией снять RAM/CPU
  сервера и убедиться, что он тянет браузер параллельно с Next.js и
  PostgreSQL. Если не тянет — вернуться к развилке «отдельный сервис».
- **Скорость.** Браузерная загрузка страницы — секунды против десятков
  миллисекунд у `fetch`. В рамках шага 1 число страниц не растёт
  (`MAX_PAGES` без изменений), поэтому в текущий синхронный запрос
  (`maxDuration = 120` в `route.ts`) укладываемся, но с меньшим запасом.
  Это ещё один довод, почему полный обход вынесен в шаг 2 (фоновый режим).
- **Playwright уже установлен** в devDependencies (`playwright` +
  Chromium 1228) в ходе проверки гипотезы. Для прода зависимость должна
  стать обычной, не dev.

## Открытые вопросы (решить до / во время реализации, не блокируют дизайн)

1. **Честный User-Agent.** Сейчас `crawl.ts:4-5` шлёт поддельный Chrome-UA.
   Для компании, аудирующей чужой комплаенс, честный UA с именем бота и
   ссылкой на себя — более защитимая позиция, но может чуть снизить
   проходимость антиботов. Решается отдельно, не входит в объём шага 1 по
   умолчанию.
2. **Точный набор маркеров детектора антибота.** Собран по одному образцу
   (KillBot/rustehnika). Cloudflare и DDoS-Guard дают другие заглушки —
   набор маркеров пополнится по мере встречи реальных сайтов; детектор
   спроектировать расширяемым.

## Критерии готовности

- На https://www.rustehnika.ru/ аудит либо показывает реальный сайт
  (браузер прошёл), либо честный статус «закрыт защитой» — но **не**
  ложное «нарушение по Google Analytics».
- Заглушка KillBot не даёт срабатывания проверки трекеров (тест на
  сохранённом HTML).
- Сайты без защиты аудируются как раньше (регресс не хуже текущего).
- Ресурсы VPS проверены и подтверждены до выката.

# Полярная звезда — инструмент аудита сайтов

Внутренний инструмент: вставил ссылку на сайт → получил доказательный аудит по 10 пунктам
чек-листа и готовое холодное письмо владельцу.

Реализует [PRD](../results/PRD.md). Состав проверок задан чек-листом на
[лендинге](../results/landing.html) и меняется **только там** — см. `src/lib/legal.ts`.

## Запуск

```bash
npm install
npm run dev     # http://localhost:3000 — вёрстка; база на VPS, локально её нет
npm run build   # производственная сборка
```

Проверить один сайт из консоли, без базы и интерфейса:

```bash
npm run check-site https://example.ru
```

Это же средство закрывает критерий приёмки из PRD §8 — прогнать контрольную выборку
из 20–30 реальных сайтов и убедиться, что ложных срабатываний нет.

## База данных

PostgreSQL живёт на VPS и слушает только `localhost` — наружу не открыт. Поэтому
`DATABASE_URL` с `localhost` корректен: приложение обращается к базе с того же сервера.
Локальный `npm run dev` до неё не достучится, и это ожидаемо: интерфейс переживает
отсутствие базы и честно об этом пишет.

```bash
# на VPS
sudo apt update && sudo apt install -y postgresql
sudo -u postgres psql -c "CREATE USER audit WITH PASSWORD '...';"
sudo -u postgres psql -c "CREATE DATABASE auditdb OWNER audit;"

echo 'DATABASE_URL=postgresql://audit:...@localhost:5432/auditdb' > .env
psql "$DATABASE_URL" -f schema.sql
psql "$DATABASE_URL" -f seed.sql   # тестовые данные, чтобы интерфейс не был пустым
```

`.env` в `.gitignore` — секреты в git не попадают.

## Как устроен аудит

`URL → crawl.ts (скачать) → checks.ts (10 проверок) → база → Word + письмо`

| Файл | Отвечает за |
|------|-------------|
| `src/lib/legal.ts` | Нормы, ссылки на КонсультантПлюс с якорями, суммы штрафов, состав 10 пунктов |
| `src/lib/crawl.ts` | Скачивание страниц, определение CMS, распознавание оболочки и подвала |
| `src/lib/checks.ts` | Сами проверки 1–10 |
| `src/lib/anglicisms.ts` | Блок Б: иностранные слова (168-ФЗ, с 01.03.2026) |
| `src/lib/letter.ts` | Холодное письмо: топ-3 нарушения |
| `src/lib/docx.ts` | Аудит-отчёт в Word |
| `src/lib/db.ts` | PostgreSQL: аудиты, находки, письма |

## Почему инструменту можно верить

Главный риск продукта (PRD §4, Риск №3) — **аудит соврёт**. Ложное обвинение убивает
доверие мгновенно. Поэтому в движок зашиты правила, а не благие намерения.

**Три исхода, четвёртого нет.** Каждая проверка возвращает ровно `violation` / `ok` /
`manual`. Тип `Verdict` не позволяет тихо потерять пункт, а `CHECK`-ограничение в
`schema.sql` — записать что-то ещё.

**Присутствие доказывается, отсутствие — нет.** Найденный счётчик, ссылка на Инстаграм
или чекбокс с `checked` — это нарушение с фрагментом кода, который можно перепроверить
руками. А вот «Политики нет» — вывод об отсутствии, и он честен, только если мы видели
то место, где ссылка обязана быть. Отсюда `canProveAbsence()`: страница отдаётся
сервером **и** подвал виден в коде. Иначе — `manual`.

**«Ничего не нашли» ≠ «всё в порядке».** Если подвала не видно, инструмент не выдаёт
справку о чистоте: `nothingFound()` вернёт `unknown`. Молча пропустить нарушение так же
недопустимо, как выдумать его.

Эти правила выросли из проверок на живых сайтах — каждое закрывает уже пойманное ложное
срабатывание:

- **Счётчик, заряженный под согласие.** У 1c-bitrix.ru код Google Tag Manager лежит в
  `<script type="text/html">` под загрузчиком `data-bx-gdpr`: браузер его не исполняет,
  пока посетитель не согласится. Обвинение «данные уже уходят» было бы ложью — теперь
  это `manual`.
- **Оболочка вместо страницы.** mvideo.ru отдаёт роботу одинаковый пустой каркас на
  любой адрес, включая заведомо несуществующий. Это давало три «нарушения» на ровном
  месте. Теперь перед обходом делается контрольный запрос по битому адресу, и всё, что
  похоже на эту заглушку, документом не считается.
- **«Политика» — это ещё и «политика».** На kp.ru ссылка «Политика» ведёт в раздел
  новостей. Инструмент принимал её за Политику конфиденциальности и выдавал справку о
  чистоте. Теперь документ опознаётся только по однозначным словам —
  «конфиденциальности», «персональных данных».
- **Подвал, которого нет в коде.** У kp.ru подвал дорисовывает скрипт, ссылок на
  документы в HTML нет вовсе. Вывод «Политики нет» был ложным — теперь такие пункты
  уходят в ручную проверку.
- **Документ называют по-разному.** «Соглашение об использовании сайта» — то же
  пользовательское соглашение. Поиск идёт регулярным выражением, а не списком подстрок.

## Чего инструмент намеренно не делает

- **Не открывает сайт в браузере.** Нет Playwright — значит, нет скриншотов и нет
  разбора того, что рисуется скриптами. Такие пункты честно уходят в ручную проверку.
  Пруф — фрагмент исходного кода: его можно сверить через «Просмотр кода страницы».
- **Проверяет размещение сервера через RDAP и ipwho.is.** DNS → IP → реестр RDAP
  (`rdap.org`), при необходимости — геобаза `ipwho.is`. Нарушение заявляется только
  если оба источника независимо подтвердили не-РФ; RDAP не ответил или назвал CDN —
  пункт 1 в этой части уходит в `manual`. Результат не выдумывается.
- **Не отправляет письма.** Письмо правится в интерфейсе и копируется — отправляет
  человек со своей почты.
- **Пункты 9 и 10 — только вручную.** Отписка проверяется в письмах рассылки, а
  уведомление РКН — в реестре. По внешнему виду сайта ни то, ни другое не видно.

## Правило про штрафы

Нормы и суммы берутся **только из КонсультантПлюс**, каждая — со ссылкой с якорем на
конкретную часть статьи. Все значения лежат в одном месте — `src/lib/legal.ts` — и
сверены с действующей редакцией ст. 13.11 КоАП РФ от 04.07.2026. Без ссылки на
первоисточник штраф в отчёт не попадает.

## Деплой

```bash
npm run build
tar --exclude=node_modules --exclude=.git --exclude=.next -czf - . \
  | ssh user@VPS_IP 'mkdir -p ~/product-mvp && tar xzf - -C ~/product-mvp'
ssh user@VPS_IP 'cd ~/product-mvp && npm install && npm run build \
  && (npx pm2 restart product-mvp || npx pm2 start npm --name product-mvp -- start)'
```

Сайт: `http://VPS_IP:3000`

> Для учебного одноразового VPS намеренно упрощено: голый HTTP, порт 3000, без
> отдельного пользователя, HTTPS и firewall. На боевом проекте так не делают.
